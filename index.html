TCP和UDP的区别

UDP的报文结构

  1. 源端口
  2. 目标端口
  3. 长度
  4. 校验和
  5. 用户数据

UDP的特点

  1. 面向非连接
    1. 传输数据时源端和终端不建立连接
  2. 不维护连接状态
    1. 无法处理丢包、乱序或重传
      3
      吞吐量只受限于数据生成效率、传输速率和机器性能
  3. 尽最大努力交付，不保证可靠交付，不需要维持复杂的链接状态表
  4. 面向报文，对应用程序提交的报文信息既不拆分也不合并
    1. 应用程序需要控制数据的大小

TCP 和 UDP 的区别

  1. 面向连接vs无连接
  2. 可靠性
  3. 有序性
  4. 速度
  5. 量级
    1. TCP——重量级，体现在报文头
    2. UDP——轻量级

TCP的四次挥手

Posted on 2024-12-14

TCP的四次挥手

标志位的含义
  • ACK的逻辑：
    • ACK用来确认接收到对方的数据，ACK=对方的SEQ+1。
  • FIN和SEQ逻辑：
    • FIN=1 表示请求关闭。
    • SEQ 是发送数据包的序列号，标识当前包的位置。
四次挥手发送的内容

第一次：FIN = 1，seq = u
第二次：ACK = 1，seq = v，ack = u + 1
第三次：FIN = 1，seq = w
第四次：ACK = 1，seq = u + 1，ack = w + 1

为什么要最后一次挥手之后还要等待2MSL：
  • 在TCP四次挥手的最后一步，客户端发送一个 ACK 来确认服务器的 FIN。
  • ==如果这个ACK丢失，服务器会重发它的FIN包==。客户端需要等待一段时间（2MSL），以确保能够接收到服务器的重发FIN并重新确认，从而避免因丢包导致的连接未正常关闭。
    原因：
  1. 确保有足够的时间让对端收到ACK包
  2. 避免新旧连接混淆
    1. 有些路由器会缓存数据包，不等待ＭＳＬ有可能导致延迟收到的数据包和新连接混在一起
为什么需要四次挥手才能断开连接

全双工通信，
发送方和接收方的都需要FIN和ACK报文

服务器出现大量CLOSED-WATI的原因：

对方关闭socket连接，我方忙于读或写，没有及时关闭连接。

  1. 检查代码，特别是资源释放的代码（程序里有bug）
  2. 检查配置，特别是处理请求的线程配置

TCP的三次握手

Posted on 2024-12-14

TCP的三次握手

传输控制协议TCP简介：

TCP全称：Transmission Control Protocol

面向连接的、可靠的、基于字节流的传输层通信协议：
  1. 面向连接：
    1. 在传输数据之前，TCP会通过“三次握手”建立一个可靠的连接。
  2. 可靠：
    1. TCP保证数据的顺序性、完整性和可靠性。
    2. 顺序性：序列号：数据包都有序号
    3. 完整性：
      1. 使用奇偶校验和函数来检验传输过程中是否有误，发送和接收时都要计算校验和
    4. 可靠性：
      1. 确认机制：对方收到则发送ACK确认
      2. 重传机制：未收到ACK则重传
  3. 基于字节流：
    1. TCP是一个字节流协议，应用层的数据传输是以字节流的形式进行的。
    2. 发送方将数据分割成小的 TCP段（Segment），并通过网络发送。
TCP报文的头部
  1. 源端口和目的端口：标识发送和接收的应用程序
    1. Source Port：2字节
    2. Destination Port：2字节
    3. 不包含IP地址信息
    4. IP地址 + 协议 + 端口号可以唯一标识网络中的一个进程，这种唯一标识叫做套接字socket
  2. 序列号：Sequence Number
    1. seq：序号，4字节
    2. 标识数据的顺序，确保数据能按顺序接收。
  3. 确认号：Acknowledge Number
    1. 接收方期望收到的下一个字节的序号
  4. 标志位：8个
    1. URG：紧急指针，为1时有效
    2. ==ACK==：确认序号标志，为1时确认号有效
    3. PSH：push标志，为1时该报文段应尽快交给应用程序而不是在缓冲区排队
    4. RST：重置连接标志
    5. ==SYN==：同步序号，用于建立连接过程
    6. ==FIN==：finish标志，用于释放连接
  5. Window：滑动窗口的大小，告诉发送方接收方的缓冲区可用空间
  6. Checksum：检验和，奇偶校验
    1. 确保数据的完整性，检测数据传输过程中是否发生了错误。

TCP三次握手的流程

在通信开始之前，TCP需要先建立一个连接，过程通过“三次握手”（Three-Way Handshake）完成。

第一次握手：

客户端发送一个 SYN（同步）包给服务器
表示客户端请求建立连接，
并且向服务器发送一个初始序列号（ISN）。

第二次握手：

服务器接收到客户端的SYN包后，
返回一个 SYN-ACK（同步-确认）包，表示同意建立连接，
并且向客户端发送服务器的初始序列号。

第三次握手：

客户端收到服务器的SYN-ACK包后，向服务器发送一个 ACK（确认）包，表示连接建立成功。
SYN：请求建立连接
ACK：确认请求和数据
seq：数据包的序列号
ack：接收方希望接收的下一个字节，确认号

为什么需要三次握手才能建立起连接

为了初始化seq Number的值（x和y）

首次握手的隐患：SYN超时

问题起因分析：

服务器收到客户端的SYN，回复SYN-ACK时没有收到ACK确认，
此时既不成功也不失败，
服务器会不断重试直到超时，
Linux默认等待63秒才断开连接

后果：

可能会使得服务器遭受SYN Flood攻击的风险
将服务器的SYN队列耗尽

对应的防护措施：

SYN队列满后，回发SYN Cookie
如果是正常连接Client会回发SYN Cookie，直接建立连接
这样即便队列满了，依然能够建立连接

建立连接后，客户端出现故障怎么办

保活机制：

像对方发送保活探测报文，
如果未收到响应则继续发送，
尝试次数达到保活探测数仍未收到响应则中断连接

TCp/IP协议

OIS的“实现”

  1. 链路层：包括物理层和数据链路层。
    1. 物理层：以二进制形式在物理媒体上传输数据。
    2. 数据链路层：传输有地址的帧以及错误检测
      1. ***ARP、RARP，MTU，PPP
  2. 网络层：为数据包选择路由
    1. ***IP，ICMP，OSPF，RIP
  3. 传输层：提供端对端的接口
    1. ***TCP，UDP
  4. 应用层：包括会话层、表示层和应用层。
    1. 会话层：解除或建立与别的节点的练习
    2. 表示层：数据格式化、代码转换、数据加解密
    3. 应用层：文件传输、电子邮件、文件服务、虚拟终端
      1. ***HTTP，FTP，Telnet

OSI七层模型

Posted on 2024-12-14

OSI七层模型

  1. 物理层：负责传输比特流
    1. 定义了物理设备的标准：如网线的类型、光纤的接口类型
    2. 网卡工作在物理层
  2. 数据链路层：负责 数据帧的可靠传输 和 节点间的物理地址识别
    1. 封装成帧与解封装
      1. 数据链路层将从网络层（第三层）收到的数据包封装成帧（Frame），并在发送数据时加上头部和尾部信息，确保数据的正确传输。
      2. 这些头部和尾部包含了传输所需的控制信息，比如目的地MAC地址、源MAC地址、帧校验序列等。
    2. 物理地址（MAC地址）识别
      1. 数据链路层通过 MAC地址（媒体访问控制地址）来唯一标识同一网络内的每个网络接口。每个网络设备的网卡（NIC）都有一个唯一的MAC地址。
      2. 在局域网（LAN）中，数据链路层通过MAC地址进行通信，决定数据应该传递到哪个设备。
    3. 错误检测与纠正
      1. 数据链路层还提供错误检测的功能。它会对传输的每一帧进行校验，通常使用循环冗余校验（CRC）来检查数据是否发生了传输错误。
      2. 如果发现错误，数据链路层会请求重新发送数据，确保接收到的数据是正确的。
  3. 网络层：负责逻辑地址分配、路径选择、数据包转发和跨网络传输
    1. 逻辑寻址与IP地址
      1. 网络层负责为网络中的每一个设备分配一个唯一的IP地址（Internet Protocol Address）。这与数据链路层的MAC地址不同，MAC地址是在局域网内唯一的物理地址，而IP地址是通过路由在不同网络间识别设备的逻辑地址。
      2. 网络层的核心任务之一就是在数据从一个网络传输到另一个网络时，根据目标IP地址来选择正确的路径。
    2. 路由选择
      1. 网络层的路由器（Router）根据==路由协议==和==网络拓扑==来确定数据包的传输路径。简单来说，路由器会根据目标IP地址来判断数据包应该传送到哪条路由上。常见的路由协议有 RIP（Routing Information Protocol）、OSPF（Open Shortest Path First）、BGP（Border Gateway Protocol）等。
      2. 路由选择可以是==静态路由==（由管理员手动配置）或==动态路由==（通过路由协议自动更新路由表）。
    3. 数据包转发
      1. 网络层会将来自上层（传输层）的数据包传送给下层（数据链路层）进行封装和发送。网络层负责处理数据包的路由转发，包括为数据包选择合适的路径，并将其发送到下一个节点（例如下一个路由器或目标设备）。
      2. 每个中间路由器会基于数据包的目标IP地址和自己的路由表来决定将数据包转发到哪条路由。
    4. 分段与重组
      1. 网络层还会负责处理==分段和重组==。当数据包的大小超过链路层的最大传输单元（MTU）时，网络层会将数据包进行分段（Fragmentation）。每个分段会加上必要的头部信息进行独立传输，并且到达目的地后，网络层会重新将这些分段进行重组（Reassembly）。
      2. 例如，在IP协议中，如果数据包的大小超过了路径上某一链路的MTU，IP层会将数据包切分为更小的片段，传输过程中每个片段都会携带一个标识，最终在接收端重新组合成完整的数据包。
  4. 传输层：负责提供端到端的可靠通信
    1. ==端到端通信==
      1. 传输层负责在网络中的两个设备（通常是计算机或网络服务器）之间建立、管理和终止通信连接。它实现了端到端的数据传输，确保数据能够从源应用程序顺利传输到目标应用程序。
    2. 数据流控制
      1. 传输层提供流量控制，确保发送方不会向接收方发送过多的数据，导致接收方来不及处理，从而避免数据丢失。它==根据接收方的处理能力来调节发送速率==。
    3. 可靠性和数据完整性
      1. 可靠传输：传输层可以提供可靠的数据传输，确保数据能够无错误地到达目标主机。例如，TCP协议会确保数据的可靠性，包括通过==重传机制来保证数据不会丢失==。
      2. 校验和：它通过校验和（Checksum）来==检查数据在传输过程中是否发生了损坏==，确保数据的完整性。
    4. ==数据分段与重组==
      1. 传输层将来自应用层的数据分段成适合下层网络传输的数据块（段）。网络层处理数据包的转发，而传输层则确保这些段能够在接收端正确重组，还可以保证数据的顺序不乱。
      2. 例如，传输层把一个大的应用数据拆分成多个小段，这些段会经过网络层传输，然后在接收端重新组合成完整的数据。
    5. 连接管理
      1. 传输层负责==端到端连接的建立、维持和拆除==。它能够处理 连接导向协议（如TCP）和无连接协议（如UDP）的不同需求。
      2. TCP（Transmission Control Protocol）是面向连接的，它在发送数据之前会建立连接，传输期间保证数据的顺序性、完整性，并通过确认机制（ACK）保证数据到达。
        1. 面向连接：传输前必须建立连接，通过三次握手协议建立可靠的连接。
        2. 可靠传输：确保数据的完整性和顺序。丢失的包会被重传。
        3. 流量控制：通过滑动窗口机制，防止发送方过快地发送数据，造成接收方的缓冲区溢出。
        4. 拥塞控制：检测网络的拥塞情况，并根据网络状态调整数据的发送速度。
      3. UDP（User Datagram Protocol）是无连接的，它不建立连接，也不保证数据的顺序或可靠传输，适合那些对速度要求高而对可靠性要求低的应用（例如视频流、在线游戏等）。
        1. 无连接：不需要建立连接，直接发送数据，速度较快，但不保证可靠性。
        2. 轻量级：UDP没有TCP的重传、排序和流量控制等机制，适合那些对时延敏感的应用，如视频会议、实时通信等。
    6. 多路复用与解复用
      1. 传输层通过端口号实现多路复用和解复用，确保不同应用程序能够共享同一个网络连接。它通过端口号将数据正确地分配给相应的应用程序。比如，当多个应用程序同时运行时，操作系统通过传输层的端口号将数据传输到正确的应用程序。
  5. 会话层：管理和控制通信会话
  6. 表示层：数据格式化、编码解码、加密和解密
  7. 应用层：为用户提供各种网络应用服务，直接与用户交互
