HTTP

HTTP基本概念

什么是HTTP

  • HTTP是超文本传输协议
  • HyperText Transfer Protocol

详细解释超文本传输协议

  • 协议：
    • 协说明有两个以上的参与者
    • 议是对参与者的一种行为约定和规范
    • HTTP是一个用在计算机世界的协议。
      • 用计算机能理解的语言
      • 确定了计算机之间交流通信的规范
      • 以及相关的各自控制和错误处理方式
  • 传输
    • HTTP是一个双向协议
    • HTTP是一个在计算机世界专门用来
      • 在两点之间传输数据
      • 的约定和规范
    • 数据虽然是在两点之间传输，但允许中间的中转或接力
  • 超文本
    • 超越了普通的文本
    • 可以是文字、图片、视频等，甚至超链接

总结：HTTP是一个在计算机世界里专门在两点之间传输文字、图片、音频、视频等超文本数据的约定和规范。

HTTP常见状态码

  • 1xx:提示信息
    • 实际用到较少
  • 2xx：成功
    • 200 OK：最常见的成功状态码
    • 204 No Content：与200基本相同，但响应头没有body数据
    • 206 Partial Content：也是成功状态，应用于HTTP分块下载或断点续传
  • 3xx：重定向
    • 301 Moved Permanently：
      • 永久重定向，请求的资源已经不存在了，需要用新的URL再次访问
    • 302 Found：
      • 临时重定向，请求的资源还在，但暂时需要用另一个URL来访问
    • 301和302都会在响应头里使用字段Location，指明后续要跳转的URL，浏览器会自动重定向新的URL
    • 304 Not Modified：
      • 不具备跳转的含义，资源未修改
      • 告诉用户可以继续使用缓冲资源
  • 4xx：客户端发送的报文有误，服务器无法处理
    • 400 Bad Request:
      • 表示客户端请求的报文有错误，但只是个笼统的错误
    • 403 Forbidden：
      • 服务端禁止访问资源，不是客户端的请求出错
    • 404 Not Found：
      • 请求的资源在服务器不存在或未找到，所以无法提供给客户端
  • 5xx：服务端错误码
    • 500：是个笼统的错误码，服务器发生了什么错误并不知道
    • 501 Not Implemented：表示客户端请求的功能还不支持
    • 502 Bad Gateway: 服务器作为网关或代理时返回的代码，表示服务器自身工作正常，访问后端服务器发生了错误
    • 503 Service Unavailable：服务器当前很忙，暂时无法响应客户端

HTTP常见字段

  • Host字段：客户端发送请求时用来指定服务器的域名
  • Content-Length：服务器返回数据时用来表明本次回应的数据长度
  • Connection 字段：客户端要求服务器使用HTTP长连接机制
    • 只要任意一端没有明确提出断开连接，则保持TCP连接
    • HTTP/1/1的版本的默认连接都是长连接，但是为了兼容老版本的HTTP，需要指定Connection首部字段的指为Keep-Alive
  • Content-Type：服务器回应时告诉客户端本次数据是什么格式
    • 客户端请求时可以用Accept字段声明自己可以接受哪些数据格式
  • Content-Encoding：说明数据的压缩方法
    • 表示服务器返回的数据使用了什么压缩格式
    • 客户端请求时用Accept-Encoding字段说明自己可以接受哪些压缩方法

Get与Post

GET和POST的区别

  • GET的语义是从服务器获取指定的资源
  • GET请求的参数一般写在URL种
  • URL规定只能支持ASCII
    • GET请求的参数只允许ASCII码
    • 浏览器对URL的长度有限制
      • HTTP协议本身对URL长度没有作任何规定
  • POST的语义是根据报文body对指定的资源做出处理
  • POST请求携带数据的位置一般在报文body中
    • body中的数据可以是任意格式，只要客户端和服务端协商好即可
    • 浏览器不对body大小做限制

GET和POST方法都是安全和幂等的吗

安全和幂等的概念

  • 安全：
    • 请求方法不会破坏服务器上的资源
  • 幂等：
    • 多次执行相同的操作，结果都是相同的
  • GET方法是安全、幂等、可被缓存的
  • POST请求不安全、不幂等，大部分实现不可缓存

实际开发中，开发者不一定按照RFC规范定义的语义来实现GET和POST，比如用GET实现增删改查，这样实现的GET方法就不是安全和幂等的；用POST方法来实现查询数据的请求，此时POST方法就是安全幂等的。

假如安全是指信息是否被泄露的话

  • 虽然POST用body传输，GET用URL传输，但并不说明GET不如POST安全
  • HTTP传输的内容都是明文的，虽然浏览器地址栏看不到POST提交的body数据，但是只要抓个包就都能看到了。
  • 要避免传输过程中数据 被窃取，就要使用HTTPS协议，这样所有的HTTP数据都会被加密传输。

GET请求可以带body吗

  • RFC规范并没有规定GET请求不能带body
  • 理论上可以，但GET的语义请求是获取请求，根据语义，不需要用到body
  • 同理，POST请求的URL也可以有参数

HTTP缓存技术

重复性HTTP请求且每次得到的数据都是一样的，就可以把请求-响应的数据缓存在本地，下次直接读取本地数据，从而提升性能
  • 缓存的两种实现方法
    • 强制缓存
      • 只要浏览器判断缓冲没有过期，就直接使用浏览器的本地缓冲
      • 决定是否使用缓存的主动性在于浏览器
      • 实现强制缓存：
        • 通过两个字段：Cache-Control和Expires,都表示资源在客户端缓存的有效期
        • 如果HTTP响应头部同时有这两个字段，那么Cache-Control的优先级要高于Expires
        • 建议使用Cache-Control字段，因为它的选项更多，设置更精细
          • 具体实现流程：
            • 浏览器第一次请求访问服务器时，服务器返回资源的同时在Response头部加上Cache-Control，设置了过期时间大小
            • 浏览器再次请求访问服务器的该资源时，会先计算该资源是否过期
              • 如果没有过期则使用缓存
              • 否则重新请求服务器
            • 服务器再次收到请求后，会再次更新Response头部的Cache-Control
    • 协商缓存
      • 由服务端告知客户是否可以使用缓存
      • 实现：
        • 第一种实现：请求头部的If-Modified-Since和响应头部的Last-Modified字段
          • Last-Modified：该响应资源的最后修改时间
          • If-Modified-Since：服务器收到请求后将其与Last-Modified进行对比，如果Last-Modified较新，说明资源被改过，返回最新资源，HTTP 200OK；否则说明资源无新修改，响应HTTP304走缓存。
        • 第二种实现：
          • 请求头部的If-None-Match和响应头部的Etag字段
            • 响应头部Etag：唯一标识响应资源
            • 具体实现：
              • 浏览器第一次请求服务器资源时，服务器返回资源并在Response头部加上ETag唯一标识
              • 浏览器再次请求服务器的资源时，先检查强制缓存是否过期
                • 没有过期则直接使用本地缓存
                • 否则在Request头部加上If-None-Match字段，该字段的值就是ETag的值
                  • 服务器收到请求后，将If-None-Match字段和ETag进行比较
                    • 若相等，返回304，不返回资源
                    • 否则返回200和资源，并在Response头部加上新的ETag唯一标识
              • 如果浏览器收到304，从本地缓存加载资源；否则更新资源
        • 前一种基于时间实现，后者基于唯一标识实现
        • Etag的优先级高于Last-Modified
          • 因为：
            • If-Modified-Since能检查到的粒度是秒级，Etag可以保证用户能在1秒内刷新多次
            • 有些服务器不能精确获取文件修改的最后时间
    • 注意：协商缓存需要配合强制缓存种的Cache-Control字段使用。
      • 当未命中强制缓存的时候，才能发起带有协商缓存字段的请求。

HTTP特性（HTTP/1.1版本）

优点

  • 简单
    • 报文格式：header+body
    • 头部信息：key-value
  • 灵活、易于拓展
    • 各类请求方法、URI/URL、状态码、头字段都没有被固定死，允许开发人员自定义和扩充
  • 跨平台

缺点

  • 无状态传输
    • 好处：减轻服务器的负担
    • 坏处：完成关联性操作时非常麻烦
      • 解决方法：Cookie技术
        • 在请求和响应报文中写入Cookie信息来控制客户端的状态
        • 在客户端第一次请求后，服务器会下发一个装有客户信息的「小贴纸」，后续客户端请求服务器的时候，带上「小贴纸」，服务器就能认得了了
  • 明文传输
    • 好处：抓包可以直接查看，方便调试
    • 坏处：容易被窃取信息
  • 不安全：
    • 明文不加密，账号信息容易泄露
    • 不验证通信方的身份，容易遭遇伪装
    • 无法证明报文完整性，信息容易被篡改

解决办法：采用HTTPS，引入SSL/TLS层。

性能

  1. 长连接
    1. 早期HTTP/1.0每发起一个请求，都要新建一次TCP连接（三次握手），而且是串行请求，增加了通信开销。
    2. HTTP/1.1提出了长连接的方式
      1. 只要任意一方没有明确提出断开连接，保持TCP连接状态
      2. 如果某个HTTP长连接超过一定时间没有任何数据交互，服务端会主动断开这个连接。
  2. 管道网络传输
    1. 同一个 TCP 连接里面，客户端可以发起多个请求，只要第一个请求发出去了，不必等其回来，就可以发第二个请求出去
    2. 但是服务器必须按照接受请求的顺序发送对应的响应
    3. 也就是说，HTTP/1.1的管道解决了请求的队头阻塞，没有解决响应的对头阻塞

注意：HTTP/1.1管道化技术不是默认开启，且浏览器基本都没有支持。

  1. 队头堵塞：当顺序发送的请求序列中的一个请求因为某种原因被阻塞时，在后面排队的所有请求也一同被阻塞了，会招致客户端一直请求不到数据

HTTPS与HTTP

区别

  1. HTTP信息是明文传输，存在安全风险
  2. HTTPS解决HTTP不安全的缺陷，在TCP和HTTP网络层之间加入了SSL/TLS安全协议，使得报文能够加密传输
  3. HTTP建立连接的过程相对简单，三次握手之后就可以传输数据，HTTPS在TCP三次握手之后，还要进行SSL/TLS的握手，才能进入加密报文传输
  4. HTTP默认端口是80,HTTPS默认端口是443
  5. HTTPS协议需要向CA申请数字证书，来保证服务器的身份是可信的

HTTPS解决了HTTP的哪些问题

  1. HTTP存在的风险
    1. 窃听
    2. 篡改
    3. 冒充
  2. HTTPS：在HTTP和TCP层之间加入SSL／TLS 协议
    1. 信息加密
    2. 校验机制
    3. 身份证书

解决窃听：混合加密

  • 对称加密和非对称加密混合：通信建立前采用非对称加密交换“会话密钥”，通信过程中使用对称加密的“会话密钥”加密明文数据。
    • 对称加密：只使用一个密钥，密钥必须保密
      • 速度快，但密钥交互不安全
    • 非对称加密：使用公钥和私钥
      • 解决了密钥交换问题但速度慢

解决篡改风险：摘要算法+数字签名

  • 通过摘要算法计算出内容的哈希值
    • 哈希值唯一
    • 无法通过哈希值导出内容
  • 但是可能内容和哈希值都被替换了，因为无法确保服务器的身份
  • 公钥加密，私钥解密：为了内容传输的安全，只有私钥能解出内容
  • 私钥加密，公钥解密：保证消息不被冒充
    • 非对称加密耗性能，不用来加密大量数据
数字签名算法：
  • 私钥加密，公钥解密
    • 私钥由服务端保管
    • 服务端向客户端颁发对应的公钥
    • 如果客户端能解密，说明是该服务器发送的
  • 不对内容加密，而是对内容的哈希值加密

数字证书

  1. 已知：哈希算法保证消息完整性，数字签名保证是持有私钥的一方发送的
  2. 万一公钥是被伪造的呢？
  3. 权威机构：CA（数字证书认证机构）
    1. 只要证书是可信的，公钥就是可信的

HTTPS如何建立连接的

SSL/TLS协议流程

  1. 客户端向服务器索要并验证服务器的公钥
  2. 双方协商生产会话密钥
  3. 双方采用会话密钥进行加密通信

TLS握手过程

  • 使用不同的密钥交换算法，TLS握手流程也不一样
    • 常用的密钥交换算法：RSA算法和ECDHE算法
详细流程
  1. ClientHello：
    1. 客户端向服务器发起加密通信请求
      1. 客户端支持的TLS协议版本
      2. 客户端生产的随机数
      3. 客户端支持的密码套件列表
    2. SeverHello
      1. 确认TLS版本，如果不支持则关闭加密通信
      2. 服务器生产的随机数
      3. 确认的密码套件
      4. 服务器的数字证书
    3. 客户端回应
      1. 先验证数字证书的真实性
      2. 用服务器公钥加密报文，发送
        1. 一个随机数
        2. 加密通信算法改变通知：之后使用会话密钥加密通信
        3. 客户端握手结束通知，同时发送之前内容的摘要供服务器校验
      3. 客户端和服务端生成本次通信的会话密钥
    4. 服务器的最后回应：
      1. 服务器接收到客户端的第三个随机数后，计算出本次的会话密钥
      2. 加密通信算法改变通知
      3. 服务器握手结束通知，同时发送之前的内容摘要
